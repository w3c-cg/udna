1. Introduction

[Diagram: Current vs DENA-enabled architecture]

Figure 1: Transition from fragmented authentication to DENA-enabled identity layer

Modern web applications struggle with fragmented identity management, relying on multiple authentication systems that create security gaps, poor user experience, and privacy concerns. This explainer explores how W3C Decentralized Identifiers (DIDs) could be integrated into application-layer networking through an approach we're calling DID-Enhanced Network Addressing (DENA).

DENA focuses on practical, incremental adoption rather than revolutionary changes. We're seeking community feedback on whether this approach addresses real problems and how it might evolve through W3C standardization.

2. User-Facing Problem

üí° The Core Problem

Users and developers face fragmented identity systems that are insecure, inconvenient, and privacy-invasive. There's no standardized way for applications to securely communicate using decentralized identities.

For End Users:

  • Password Hell: Average user manages 100+ passwords
  • Privacy Erosion: Services track users across applications
  • Account Lock-in: Can't move identity between services
  • Recovery Nightmares: Lose access when email/phone changes

For Developers:

  • Authentication Complexity: Months spent on auth systems
  • Vendor Lock-in: Dependent on Google/Apple/Microsoft auth
  • Service Communication: No standard way for apps to talk securely
  • Compliance Burden: Different regulations per region

Motivating Use Cases

Use Case 1: Cross-Application Single Sign-On

A user logs into their healthcare portal and wants to securely share records with their insurance app and pharmacy app without creating new accounts.

Current pain: Create 3 separate accounts, share passwords or use insecure links

DENA potential: One DID authenticates to all three services with permission controls

Use Case 2: Service-to-Service Communication

A weather app wants to send severe alerts to a user's smart home system and calendar app.

Current pain: No standard way for services to discover and authenticate each other

DENA potential: Services use DIDs to find and securely message each other

Use Case 3: Privacy-Preserving Analytics

Multiple research institutions want to collaborate on data without exposing individual user identities.

Current pain: Either share raw data or build complex anonymization

DENA potential: Pairwise DIDs prevent correlation while allowing verification

2.1 Goals

Goal 1: User Control & Privacy

Users should control their digital identity without centralized intermediaries, with built-in privacy protections.

Goal 2: Developer Simplicity

Make identity integration as simple as including a JavaScript library, not building complex auth systems.

Goal 3: Backward Compatibility

Work alongside existing systems (OAuth, cookies) with clear migration paths.

Goal 4: Performance

Add minimal latency (target: <100ms for auth, <5ms for message verification).

Goal 5: Standards-Based

Build on W3C DIDs, Verifiable Credentials, and existing web standards.

2.2 Non-goals

Non-goal 1: Replace IP or DNS

DENA works at application layer, not network layer.

Non-goal 2: New Browser APIs (initially)

Start with JavaScript libraries, consider native integration later.

Non-goal 3: Solve All Identity Problems

Focus on application-layer identity, not legal identity or KYC.

Non-goal 4: Require Blockchain

Support DID methods like did:web and did:key that don't need blockchain.

3.2 Core Concepts

1. DID as Application Address

Instead of just identifying users, DIDs can address services and APIs:

// DID addressing example
const chatService = 'did:web:api.example.com:services:chat';
const userProfile = 'did:web:social.example.com:users:alice#profile';
const paymentAPI = 'did:web:pay.example.com:apis:v2';

2. Service Discovery via DID Documents

DID documents describe what services are available and how to access them:

// Example DID document with services
{
  "id": "did:web:api.example.com",
  "service": [
    {
      "id": "#chat-api",
      "type": "DENAHttpEndpoint",
      "serviceEndpoint": "https://api.example.com/chat",
      "supportedProtocols": ["DENA-Messaging-v1", "DIDComm-v2"]
    },
    {
      "id": "#auth-endpoint",
      "type": "DENAAuthEndpoint", 
      "serviceEndpoint": "https://auth.example.com/did"
    }
  ]
}

3.3 Integration Patterns

Pattern Description When to Use
1. DID-Enhanced OAuth Add DID support to existing OAuth flows Gradual migration, existing OAuth systems
2. Pure DENA Full DID-based authentication and messaging New applications, maximum privacy
3. Gateway Pattern DENA gateway translates between systems Enterprise, legacy system integration
4. Hybrid Approach Mix traditional and DENA auth Transition periods, user choice

4. Alternatives Considered

Alternative 1: Extend OAuth 2.1/OpenID Connect

Why not chosen as primary: Still relies on centralized identity providers, doesn't solve user control or service-to-service communication.

Alternative 2: WebAuthn Everywhere

Why not chosen as primary: Excellent for authentication but doesn't provide portable identity or service discovery capabilities.

Alternative 3: Existing Decentralized Protocols (ActivityPub, Matrix)

Why not chosen as primary: These solve different problems (social networking, chat) rather than general application identity layer.

Alternative 4: Do Nothing (Current Ecosystem)

Why not chosen: Current fragmentation causes real security, privacy, and developer experience problems that users and developers complain about daily.

5. Accessibility, Internationalization, Privacy, and Security Considerations

‚ö†Ô∏è Critical Consideration: Key Management

How do non-technical users manage cryptographic keys? DENA must consider:

  • Hardware security keys (Yubikey, etc.)
  • Cloud key management services
  • Social recovery mechanisms
  • Emergency access procedures

Accessibility

  • Screen Reader Support: All DENA UIs must work with assistive technologies
  • Cognitive Load: Key management must be understandable by non-experts
  • Error Recovery: Clear, simple recovery flows for lost credentials

Internationalization

  • Name Systems: Support for non-Latin scripts in identifiers
  • Legal Identity: Different countries have different identity requirements
  • Time Zones: Timestamp handling for global systems

Privacy

  • Pairwise DIDs: Different DIDs per relationship to prevent correlation
  • Data Minimization: Only request necessary identity attributes
  • GDPR Compliance: Right to deletion, data portability

Security

  • Cryptographic Agility: Ability to upgrade algorithms as needed
  • Phishing Protection: UX patterns that prevent credential theft
  • Audit Logging: Comprehensive logs for security incidents

Prototype Code Examples

// Simple DENA client usage
import { DENAClient } from '@dena/sdk';

const client = new DENAClient({
  did: 'did:web:example.com:users:alice',
  resolver: 'https://resolver.dena.dev'
});

// Authenticate to a service
await client.authenticate('did:web:service.example.com');

// Send a secure message
const response = await client.sendMessage({
  to: 'did:web:service.example.com:api',
  type: 'data-request',
  body: { query: 'user-profile' }
});
