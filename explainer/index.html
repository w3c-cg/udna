Introduction

Modern web applications face significant challenges with identity management, authentication, and secure communication between services. This document explores how W3C Decentralized Identifiers (DIDs) might be integrated into application-layer networking to address these challenges while maintaining compatibility with existing web infrastructure.

We're seeking community input on problem definition, use cases, and potential approaches before proposing any specific solutions.

User-Facing Problem

End users experience several identity-related problems when using web applications:

  • Password fatigue: Users must remember and manage numerous passwords across different services
  • Privacy erosion: Services can track users across different applications and contexts
  • Security fragmentation: Different applications implement authentication differently, creating inconsistent security experiences
  • Account recovery challenges: Users lose access when email accounts or phone numbers change

Application developers face complementary problems:

  • Authentication complexity: Implementing secure authentication requires significant expertise and ongoing maintenance
  • Dependency risks: Reliance on third-party identity providers creates single points of failure
  • Interoperability gaps: Applications cannot securely communicate without shared identity infrastructure
  • User experience fragmentation: Different authentication flows across applications confuse users

Goals

Goal 1: Enable users to control their digital identity across applications without centralized intermediaries
Goal 2: Allow applications to securely authenticate users and services using open standards
Goal 3: Enable privacy-preserving communication between applications and services
Goal 4: Maintain backward compatibility with existing web infrastructure
Goal 5: Provide practical migration paths for existing applications

Non-goals

Non-goal 1: Replacing IP addresses or DNS infrastructure
Non-goal 2: Creating a new transport-layer protocol
Non-goal 3: Requiring browser vendors to implement specific features (initially)
Non-goal 4: Solving all identity problems across all contexts

Proposed Approach

We've been exploring several potential approaches and seek community feedback on their viability:

Option A: DID-Based Authentication Enhancement

Extending existing authentication mechanisms (OAuth, OpenID Connect) with DID support:

// Conceptual example - not a proposal
async function authenticateWithDID(did, options) {
    // Resolve DID to get authentication endpoints
    const didDocument = await resolveDID(did);
    const authEndpoint = findAuthEndpoint(didDocument);
    
    // Perform authentication flow
    return await performDIDAuth(authEndpoint, options);
}

Option B: Application-Layer DID Addressing

Using DIDs as application-layer addresses for secure messaging and API calls:

// Conceptual example - exploring patterns
const message = {
    to: 'did:web:api.example.com:services:chat',
    from: 'did:web:client.example.com:users:alice',
    body: 'Hello World',
    // Cryptographic signatures would be required
};

Option C: Hybrid Approach

Combining existing authentication with DID-enhanced features gradually.

Key Questions for Community Discussion:

  • Which integration patterns provide the best balance of security and usability?
  • How can we ensure backward compatibility during transition?
  • What existing standards (DIDComm, VC-DATA-MODEL) should we build upon?
  • What performance considerations are most important for adoption?
  • How do we address key management challenges for end users?

Alternatives Considered

Alternative 1: WebAuthn-Only Approach

Using WebAuthn for authentication without DIDs. This provides strong authentication but doesn't solve identity portability or service-to-service authentication challenges.

Alternative 2: OAuth 2.1 with Extensions

Extending OAuth 2.1 with new grant types and claims. This builds on familiar infrastructure but may not provide the same level of user control as decentralized approaches.

Alternative 3: Proprietary Identity Systems

Many platforms offer proprietary identity solutions (Sign in with Apple, Google Identity Services). These provide good user experience but create vendor lock-in and centralization.

Open Questions:

  • Should we focus on incremental improvements to existing systems?
  • Is now the right time for more fundamental changes to web identity?
  • How do we balance innovation with practical deployment concerns?

Accessibility, Internationalization, Privacy, and Security Considerations

Accessibility

Any solution must consider users with disabilities. Key questions:

  • How do users with cognitive disabilities manage cryptographic keys?
  • Are authentication flows compatible with screen readers and other assistive technologies?
  • How do we prevent excluding users with limited technical proficiency?

Internationalization

Identity systems must work globally:

  • How do different legal jurisdictions affect identity verification?
  • What localization challenges exist for recovery mechanisms?
  • How do we handle diverse naming conventions and identification documents?

Privacy

Core to the decentralized identity vision:

  • How do we prevent correlation across different services?
  • What data minimization principles should guide design?
  • How do users exercise rights like deletion and portability?

Security

Critical for identity systems:

  • How do we handle key recovery without compromising security?
  • What are the attack surfaces for application-layer identity?
  • How do we prevent phishing and social engineering attacks?
